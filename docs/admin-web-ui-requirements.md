# 管理用 Web UI 要件定義

## 文書の位置づけ

Slack ログ同期（GAS）の**運用・監視・限定的な設定変更**を、設定スプレッドシートを直接編集せずに行うための **Web アプリ（Google Apps Script Web アプリ）** に関する要件を定義する。

- **同期ワーカーのアルゴリズム・シート列定義の正本:** [slack-log-sync-design-proposal.md](./slack-log-sync-design-proposal.md)
- **本書のスコープ外:** リポジトリのディレクトリ構成、clasp の運用、同期ワーカー実装の詳細（別資料・README に委ねる）。

**ステータス:** 要件定義（運用方針・MVP 範囲の主要項目は本文で確定）。列追加・Slack API 呼び出し詳細は実装・設計書追随で確定する。

---

## 1. 背景・目的

### 1.1 背景

- チャンネルごとの同期状態はスプレッドシート（想定名: `slack_log_to_drive_setting`）の `channel_sync_state` 等で管理される。
- 列数が多く、ステータス種別・日時・カーソル・ロック・エラー情報が混在するため、**シートを直接開いて編集する運用は負荷が高く、誤入力・列ずれのリスク**がある。

### 1.2 目的

- 運用担当が **ブラウザから安全に一覧・検索・閲覧**できるようにする。
- **運用で必要な範囲の項目のみ**を、入力支援付きで更新できるようにする（同期ワーカーが解釈する値の整合性を UI 側で担保しやすくする）。
- **組織内ユーザーが URL から新規チャンネルを登録**できるようにする（Slack 名から ID 解決まで UI 側で完結させる）。
- スプレッドシートの**生データ編集を日常運用から切り離す**（緊急時の直接編集は設計上禁止しないが、標準動線は Web UI とする）。

### 1.3 成功の目安（非機能とあわせたゴール像）

- 同期対象チャンネルの状態を、**シートを開かずに**把握できる。
- `DISABLED` 切り替えや `note` 更新など、**定義された操作以外は UI から行えない**（または権限・確認で抑止される）。
- Slack Bot Token 等のシークレットは **ブラウザに渡らない**。
- **Web UI 上の更新時刻**は `ui_last_updated_at` 等で追える（§3.2）。**来訪者個人の識別は行わない**運用も可（§3.1）。

## 3. 前提（システム上の位置づけ）

| 項目 | 内容 |
|------|------|
| デプロイ | **同期ワーカー用 GAS とは別プロジェクト**のスタンドアロン GAS として Web アプリを公開する。 |
| データアクセス | 設定スプレッドシート等は **サーバー側 GAS**（`SpreadsheetApp` 等）から読み書きする。クライアントは `google.script.run` 等でサーバー関数を呼び出す。 |
| シークレット | Slack Token、スプレッドシート ID 等は **Script Properties**（または GAS 側の安全な設定）に保持し、**HTML/クライアント JS に埋め込まない**。新規チャンネル登録用に、管理 UI 用 GAS にも **Slack Bot Token**（または同等の Bot 用認証情報）を Script Properties で保持し、**サーバー側のみ**で Slack API を呼び出す。 |

### 3.1 スプレッドシート共有・実行ユーザー（確定）

- **運用方針（現在）:** Web アプリは **自分（デプロイした Google アカウント）として実行**する。スプレッドシートへの読み書きは **常にそのアカウントの権限**で行う。
- **スプレッドシート共有:** デプロイ担当者に編集権限があればよい。**来訪者にシートを共有する必要はない**。
- **エンドユーザー**は URL から UI にアクセスするが、シート上の変更はすべて **デプロイ担当者名義の権限**による GAS 実行として行われる。

### 3.2 Web UI 操作の記録（確定・方針変更）

- **来訪者個人のメールは取得・記録しない**（`Session.getActiveUser()` に依存した操作者ログは求めない）。
- **日時:** Web UI から `channel_sync_state` を更新したときは **`ui_last_updated_at`** にサーバー時刻を書き込む（新規チャンネル登録・§5.3 の保存が対象）。
- **`ui_last_updated_by`:** **空のまま**にするか、**常にデプロイ担当者**（`Session.getEffectiveUser().getEmail()` 等）を書き込むか、実装で統一する。いずれも「誰がその操作をしたか」の区別には使えない。
- **列の扱い:** 設計書 §4.1・同期ワーカー `Config.js` と**列位置を揃える**。同期ワーカーは当該列を**読み書きしない**。
- **`admin_ui_audit`:** 任意。採用する場合も **actor は上記方針**（個人特定しない）に合わせる。

## 4. 利用者・認証・認可

### 4.1 利用者

- **利用者:** 同一 Google Workspace（組織）内のユーザーで、**Web アプリの URL を知っている者**。同期の監視・チャンネル登録・優先度調整・無効化などを行う。
- **想定:** URL は組織内共有とし、**ログイン済み Google アカウントさえあれば利用可能**とする（許可メールの事前登録は不要）。

### 4.2 認証（確定）

- **Google アカウントによるアクセス**を必須とする。GAS Web アプリのデプロイは **「組織内（同一ドメイン）のみ」**等、組織外匿名アクセスにならない設定とする。
- **メールアドレスによる許可リストは採用しない**（必須としない）。アクセス制御は「Workspace に属するユーザー」＋「URL の周知範囲」に委ねる。

### 4.3 認可

- 初期リリースでは **ロール分離（管理者／閲覧のみ等）は必須としない**（単一ロールでよい）。将来、閲覧専用ロールを追加する余地は残す。

### 4.4 セキュリティ上の注意

- URL 秘匿のみでは強いセキュリティにはならない。**Workspace の認証境界**と **Slack Token のサーバー側保管**に依存する。より厳格な制限が必要になった場合は、許可リストや IAP 等を別途検討する。

## 5. 機能要件

### 5.1 データ範囲（参照の正）

- **一覧・詳細の主対象:** `channel_sync_state` シートの行（1 行 = 1 チャンネル）。列の意味・値域は設計書 [slack-log-sync-design-proposal.md](./slack-log-sync-design-proposal.md) の「§4. `channel_sync_state` シート列」に従う。§3.2 の監査列を追加する場合は設計書・`Config.js` を更新する。
- **`thread_queue`:** MVP では**一覧の参照のみ**または**非表示**とする（フル CRUD は後続）。必要なら件数サマリ等は別途検討。
- **`slack_user_cache`:** MVP では **UI からの編集対象外**（同期ワーカーが更新する）。

### 5.2 画面構成（必須）

| ID | 画面 / 機能 | 要件 |
|----|-------------|------|
| S-01 | **チャンネル一覧** | `channel_sync_state` の全行（またはページ分割）を表形式で表示する。 |
| S-02 | **フィルタ・ソート** | 少なくとも `status` での絞り込み、および「エラーあり」（例: `last_error_at` が空でない、または `status === ERROR`）の絞り込みができること。ソートキーは `channel_name_current` / `live_last_message_at` / `sort_last_run_at` から MVP で選択してよい。 |
| S-03 | **チャンネル詳細** | 一覧から 1 チャンネルを選び、当該行の**全列**（または運用に必要な列を明示したサブセット）を表示する。設計書上の意味が分かりにくい列には UI ラベルまたはツールヒントで補足する。 |
| S-04 | **手動更新** | §5.3 で許可された列のみ、フォーム等で編集し保存できる。保存時はバリデーション（値域・日時形式・列名整合）を行う。保存時に §3.2 の監査情報を更新する。 |
| S-05 | **フィードバック** | 保存成功・失敗をユーザーに明示する。失敗時は可能な範囲で理由（権限、競合、バリデーション）を表示する。 |
| S-06 | **新規チャンネル登録（MVP・重要度高）** | §5.4 に従う。 |

### 5.3 更新可能列と `status` 許可遷移（確定）

同期ワーカーが解釈する `status` は `Config.js` の `STATUS`（`PENDING` / `RUNNING` / `WAITING` / `ERROR` / `DISABLED`）に合わせる。空文字は未設定扱いとし、**新規行の初期値はサーバーが `PENDING`（または運用合意の値）をセット**する。

#### 5.3.1 列ごとの扱い

| 列名 | 扱い | 備考 |
|------|------|------|
| `status` | **編集可（許可遷移のみ）** | 下表にない遷移は UI で選択不可。 |
| `priority_interrupt_at` | **編集可** | 日時のクリア・設定。 |
| `note` | **編集可** | 自由記述。 |
| `channel_name_current` | **読み取り専用** | 表示用。新規登録時は Slack API で解決した名称で初期化。ワーカー実行時も最新化されうる。 |
| `lock_owner`, `lock_until` | **読み取り専用** | 詳細画面で表示するが UI から更新しない。 |
| その他（カーソル・fileId・`registered_at` 等） | **読み取り専用**（新規行はサーバー初期化） | 人為の直接編集でワーカーと不整合になりやすいため。 |

#### 5.3.2 `status` の UI からの許可遷移（確定）

**`RUNNING` の間は `status` を UI から変更しない**（実行中の行は表示のみ。ワーカーのロック・再開ロジックと競合させない）。

| 遷移元 `status` | UI から設定可能な遷移先 |
|-----------------|-------------------------|
| `PENDING` | `DISABLED` |
| `WAITING` | `DISABLED` |
| `ERROR` | `PENDING`, `WAITING`, `DISABLED`（再試行の待機列へ戻す、または停止） |
| `DISABLED` | `PENDING`, `WAITING`（同期の再開） |
| `RUNNING` | （なし） |
| 空（未設定） | 詳細画面での手動変更は想定しない。新規行はサーバーが初期化する。 |

上記以外の遷移（例: 任意の状態から `ERROR` への手動設定、`RUNNING` への手動設定）は **UI では禁止**。

### 5.4 新規チャンネル登録（MVP・確定）

**優先度: 高。** MVP に含める。

1. ユーザーが **Slack チャンネル名**を入力する（表示名の表記ゆれに対応するため、**完全一致に加え、先頭の `#` の有無を吸収**する等の正規化を行う。具体ルールは実装で定める）。
2. **サーバー側**で Slack Web API を呼び出し、**当該ワークスペース内で Bot が参照可能な範囲**でチャンネルを特定し **`channel_id`** を取得する。
3. `channel_sync_state` を **`channel_id` で検索**し、**既に行が存在する場合は登録せず**、既存行である旨をユーザーに返す。
4. 存在しない場合、**新規行を追加**する。初期値は設計書および同期ワーカーが期待する形に合わせる（例: `status` は `PENDING`、カーソル・fileId・日時系は空または初期値、`registered_at` / `registered_by` は **サーバーが自動設定**。`registered_by` は操作者メールではなく **デプロイ担当者メールまたはシステムラベル**等で統一してよい（§3.2）。
5. Slack API で名前が複数ヒット・未解決・Bot が見えないチャンネル等の場合は、**エラーメッセージで理由を返す**。
6. 本フロー完了時も §3.2 の **`ui_last_updated_at`**（および方針に従う `ui_last_updated_by`）を更新する。任意の監査シートがあれば同方針で追記する。

**Slack API の扱い:** 同期ワーカー用 Token と同一 Bot を流用してよいが、**Script Properties は管理 UI 用 GAS プロジェクト側**に設定する。レート制限・スコープ（`channels:read` / `groups:read` 等）は実装時に Bot の参加チャンネル範囲と照合する。

### 5.5 MVP で行わないこと（再掲）

- `thread_queue` のフル CRUD。
- Drive 上の CSV/JSONL の**本文**の表示・ダウンロード（リンク表示のみは将来検討可）。

## 6. 非機能要件

| ID | 項目 | 要件 |
|----|------|------|
| NF-01 | セキュリティ | シークレットをクライアントに渡さない（§3）。HTTPS で配信される GAS Web アプリの標準に従う。 |
| NF-02 | 性能・制限 | GAS の 1 回実行時間・回数制限内で完結する設計とする。一覧は**ページング**または**最大行数上限**を設け、全件一括読み込みでタイムアウトしないこと。新規登録時の Slack API は必要最小限の呼び出しとする。 |
| NF-03 | 同時更新 | 同一行をシート上で直接編集された場合と Web UI から保存した場合の**最終書き勝ち**は許容する。必要なら保存前に `sort_last_run_at` 等の簡易チェックを将来検討（MVP では必須としない）。 |
| NF-04 | 可用性 | GAS / Google の障害・メンテナンスに依存する。独自 SLA は定めない。 |
| NF-05 | 言語・表示 | UI 文言は**日本語**を主とする。日時は運用者が理解しやすい形式（タイムゾーン方針は実装時に固定して文書化）。 |
| NF-06 | 保守 | 列名は設計書・同期ワーカーの `Config.js` と**一致させる**こと。監査列・監査シートを追加する場合はワーカーと同時に列定義を更新する。 |
| NF-07 | 記録 | §3.2（日時の記録、個人操作者は記録しない方針）に合わせる。 |

## 7. UI / 実装方針（技術レベル）

- **HTML Service** による Web アプリ、`doGet` で HTML を返す構成とする。
- データ取得・更新・Slack API 呼び出しは **`google.script.run` によるサーバー関数**経由とする（クライアントのみで完結する処理は最小限）。
- 見た目のフレームワーク・コンポーネントは**必須としない**（実装フェーズで選定）。アクセシビリティは可能な範囲で考慮するが、WCAG 準拠を MVP で必須とはしない。
