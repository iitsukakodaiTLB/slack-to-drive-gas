# 管理用 Web UI 要件定義（たたき）

Slack ログ同期（GAS）の**運用・監視・軽い設定変更**を、スプレッドシートを直接開かずに行うための Web アプリの要件ドラフトです。  
同期ロジック本体の設計は [slack-log-sync-design-proposal.md](./slack-log-sync-design-proposal.md) を正とします。

**ステータス:** たたき（Web UI の機能・画面は未合意）。**リポジトリ構成・clasp 運用は §3 で確定済み。** Web UI の実装は機能要件の合意後に進める。

---

## 1. 背景・目的

- 設定・状態はスプレッドシート `slack_log_to_drive_setting`（想定）上の `channel_sync_state` 等で管理される。
- 列数・状態が多く、**運用者がシートを直接編集する負担・誤操作リスク**を下げたい。
- 一覧・フィルタ・安全な更新 UI を Web で提供する。

---

## 2. アーキテクチャ方針（チャット上の合意の整理）

| 項目 | 方針 |
|------|------|
| デプロイ単位 | **同期ワーカー用 GAS と別の、スタンドアロン GAS プロジェクト**とする。 |
| 理由 | デプロイ・権限・実行トリガーを分離し、管理 UI の変更が同期処理に波及しにくくする。 |
| データの所在 | 同一スプレッドシート（および必要なら同一 Drive）を、サーバー側 GAS（`SpreadsheetApp` 等）から参照・更新する想定。クライアントに Slack Token を渡さない。 |

**未確定（要決定）:** 管理 UI 用 GAS がスプレッドシートにアクセスするためのサービスアカウント／実行ユーザーと、スプレッドシートの共有設定。

---

## 3. リポジトリ・clasp（確定）

- **ディレクトリ:** ルート配下に `gas/slack-sync-worker/`（同期ワーカー）と `gas/admin-web-ui/`（管理 Web UI）を置く。
- **clasp の運用:** 各 GAS ごとに **`gas/...` 直下へ `cd` してから** `clasp push` / `pull` / `open` / `logs` を実行する（`.clasp.json` は各フォルダ内に置き、`rootDir` は通常不要）。
- **ルートからの代替:** リポジトリルートの `package.json` に `clasp:push:sync` / `clasp:push:admin` 等の npm script を定義し、内部で上記ディレクトリに `cd` して同じ操作を行う。詳細は [../README.md](../README.md) と [../gas/README.md](../gas/README.md)。
- **`.clasp.json` の Git 管理:** 本リポジトリは**プライベートかつ個人開発**のため、各 `.clasp.json`（`scriptId` 含む）を **Git に含めてよい**方針とする。チーム公開やフォークを想定する場合は `.clasp.json.example` のみコミットする運用へ切り替えることを検討する。

---

## 4. 利用者・認証（たたき）

- **想定利用者:** ワークスペース内の運用担当（Google アカウント）。
- **認証の候補（要選定）:**
  - GAS Web アプリの「自分で実行」相当で、Google ログイン済みユーザーのみアクセス可能にする。
  - さらに **メールアドレスの許可リスト**（Script Properties または設定シート）で制限する。
- **非目標（初期）:** 一般公開、Slack OAuth でのエンドユーザー向けログイン。

---

## 5. 機能要件（MVP 候補）

同期設計書の `channel_sync_state` / `thread_queue` / エラー列を踏まえた**最小価値**の案。

| ID | 要件（案） | 備考 |
|----|------------|------|
| F-01 | チャンネル同期状態の**一覧表示**（`channel_id`, `channel_name_current`, `status`, 主要な日時・エラー要約など） | 列の選定は MVP で絞る |
| F-02 | **フィルタ**（例: `status`, エラーありのみ） | |
| F-03 | **手動更新**（例: `DISABLED` の切り替え、`note` の編集、優先度関連列の更新） | 同期ワーカーと競合しない更新順・ロック方針を設計書と照合すること |
| F-04 | 単一チャンネルの**詳細表示**（全列または運用に必要な列） | |
| F-05 | **読み取り専用**の実行サマリ（最終成功/失敗時刻、連続失敗回数の表示） | 書き込みは F-03 に含めてもよい |

**MVP に含めない候補（明示的に後回し可）:**

- `thread_queue` のフル CRUD UI
- Slack API を管理 UI から直接呼ぶ機能（チャンネル探索など）
- ログファイル本体の閲覧・ダウンロード（Drive リンク程度なら検討余地）
- 監査ログ・複数人同時編集の高度な制御

---

## 6. 非機能要件（たたき）

| ID | 項目 | 内容（案） |
|----|------|------------|
| NF-01 | セキュリティ | シークレットは Script Properties のみ。クライアント JS にトークンを埋め込まない。 |
| NF-02 | 性能 | GAS 実行時間制限内。一覧はページングまたは行数上限。 |
| NF-03 | 保守性 | 同期ワーカーと**列定義の単一情報源**をどう保つか（共通定数ファイルの共有は別 GAS では困難 → ドキュメントまたは手動同期で許容するか要判断）。 |

---

## 7. UI / 技術スタック（たたき）

- **GAS Web アプリ**（`doGet` + HTML Service）。  
- クライアントは **サーバー関数 `google.script.run`** 経由でデータ取得・更新する形を基本とする（簡素さ優先）。
- **未確定:** CSS フレームワークの有無、スプレッドシート風テーブルライブラリの採用。

---

## 8. オープン課題（合意が必要な論点）

1. MVP に含める **更新可能な列** の範囲（`lock_*` を UI から触らせるか等）。
2. 認証は **Google のみ**か、**許可リスト**まで必須か。
3. ~~リポジトリの **ディレクトリ構成・clasp 運用** の最終決定。~~（§3 で確定済み）
4. 管理 UI 用 GAS の **デプロイ権限**（誰がデプロイし、どの Google アカウントで実行するか）。
5. 同期ワーカーとの **列定義の同期**方針（ドキュメント管理でよいか）。
